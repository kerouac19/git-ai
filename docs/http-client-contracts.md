# Rust Client HTTP Contracts

This document is derived only from Rust client code under `src/`. It does not
use `server/`, `server-go/`, or server-side documentation. CI workflow YAML
templates are also excluded because they are not Rust client code.

The Rust client currently issues `GET` and `POST` requests only. No `PUT`,
`PATCH`, or `DELETE` HTTP calls were found in `src/**/*.rs`.

Unless noted otherwise, product API paths are resolved against
`Config::api_base_url`, whose default is `https://usegitai.com`.

## Common Product API Headers

Requests made through `ApiContext` include:

- `User-Agent: git-ai/<CARGO_PKG_VERSION>`
- `X-Distinct-ID: <config distinct id>`
- `X-API-Key: <api key>` when configured
- `X-Author-Identity: <git author identity>` when `X-API-Key` is configured and git identity resolves
- `Authorization: Bearer <access token>` when an OAuth access token is available

JSON `POST` requests include:

- `Content-Type: application/json`

## OAuth

### Start Device Flow

- Method: `POST`
- Path: `/worker/oauth/device/code`
- Body:

```json
{}
```

- Success response:

```ts
{
  device_code: string,
  user_code: string,
  verification_uri: string,
  verification_uri_complete?: string,
  expires_in: number,
  interval: number
}
```

### Poll Device Token

- Method: `POST`
- Path: `/worker/oauth/token`
- Body:

```ts
{
  grant_type: "urn:ietf:params:oauth:grant-type:device_code",
  device_code: string,
  client_id: "git-ai-cli"
}
```

- Success response:

```ts
{
  access_token: string,
  token_type: string,
  expires_in: number,
  refresh_token: string,
  refresh_expires_in: number
}
```

- Error response:

```ts
{
  error: string,
  error_description?: string
}
```

The client explicitly handles `authorization_pending`, `slow_down`,
`access_denied`, and `expired_token`.

### Refresh Access Token

- Method: `POST`
- Path: `/worker/oauth/token`
- Body:

```ts
{
  grant_type: "refresh_token",
  refresh_token: string,
  client_id: "git-ai-cli"
}
```

- Success and error responses are the same as Poll Device Token.

### Exchange Install Nonce

- Method: `POST`
- Path: `/worker/oauth/token`
- Body:

```ts
{
  grant_type: "install_nonce",
  install_nonce: string,
  client_id: "git-ai-cli"
}
```

- Success and error responses are the same as Poll Device Token.

## Bundles

### Create Bundle

- Method: `POST`
- Path: `/api/bundles`
- Auth: sent through `ApiContext`; a stored OAuth access token is attached as
  `Authorization: Bearer <access token>` when available.
- Body:

```ts
{
  title: string,
  data: {
    prompts: {
      [prompt_id: string]: {
        agent_id: {
          tool: string,
          id: string,
          model: string
        },
        human_author?: string,
        messages: Message[],
        total_additions: number,
        total_deletions: number,
        accepted_lines: number,
        overriden_lines: number,
        messages_url?: string,
        custom_attributes?: { [key: string]: string }
      }
    },
    files?: {
      [path: string]: {
        annotations: {
          [prompt_hash: string]: Array<number | [number, number]>
        },
        diff: string,
        base_content: string
      }
    }
  }
}
```

`Message` is serialized as a tagged enum with `type` in `snake_case`:

```ts
type Message =
  | { type: "user", text: string, timestamp?: string }
  | { type: "assistant", text: string, timestamp?: string }
  | { type: "thinking", text: string, timestamp?: string }
  | { type: "plan", text: string, timestamp?: string }
  | { type: "tool_use", name: string, input: unknown, timestamp?: string };
```

- Success response:

```ts
{
  success: boolean,
  id: string,
  url: string
}
```

- Error response for 400/500:

```ts
{
  error: string,
  details?: unknown
}
```

## CAS

### Upload CAS Objects

- Method: `POST`
- Path: `/worker/cas/upload`
- Body:

```ts
{
  objects: Array<{
    content: unknown,
    hash: string,
    metadata?: { [key: string]: string }
  }>
}
```

- Success response:

```ts
{
  results: Array<{
    hash: string,
    status: string,
    error?: string
  }>,
  success_count: number,
  failure_count: number
}
```

- Error response for 400/500:

```ts
{
  error: string,
  details?: unknown
}
```

### Read CAS Objects

- Method: `GET`
- Path: `/worker/cas/`
- Query string:

```ts
{
  hashes: string
}
```

`hashes` is a comma-joined list generated by `hashes.join(",")`. The client
does not URL-encode the hash list.

- Success response:

```ts
{
  results: Array<{
    hash: string,
    status: string,
    content?: unknown,
    error?: string
  }>,
  success_count: number,
  failure_count: number
}
```

A 404 response is treated as:

```ts
{
  results: [],
  success_count: 0,
  failure_count: <requested hash count>
}
```

## Notes (HTTP Authorship Notes Backend)

Opt-in via `notes_backend.kind = http` and `notes_backend.backend_url =
<base>` in `~/.git-ai/config.json`. Default kind is `git_notes` and the
endpoints below are not called. The backend may share the main API host or
run as a separate service; auth still uses the standard `ApiContext` headers
(`X-API-Key` or `Authorization: Bearer <jwt>` plus `User-Agent`,
`X-Distinct-ID`, optional `X-Author-Identity`). Both endpoints are skipped
when neither `is_logged_in()` nor `has_api_key()` is true (matches the daemon
flush_notes pattern). See `src/api/notes.rs` and the reference implementation
in `src/notes/reference_server.rs`.

### Upload Notes

- Method: `POST`
- Path: `/worker/notes/upload`
- Body:

```ts
{
  entries: Array<{
    commit_sha: string,  // hex commit SHA
    content: string      // serialized authorship-log payload
  }>
}
```

- Success response (200):

```ts
{
  success_count: number,
  failure_count: number
}
```

- Error response (400) for malformed body:

```ts
{
  error: string,
  details?: unknown
}
```

The reference server caps request body size at 50 MiB; production backends
should enforce a comparable cap. There is no client-side per-request entry
limit on the upload path (the bulk uploader `git-ai notes migrate` chunks
locally), so the server should set its own batch ceiling.

### Read Notes

- Method: `GET`
- Path: `/worker/notes/`
- Query string:

```ts
{
  commits: string  // comma-joined hex SHAs
}
```

`commits` is built by `commit_shas.join(",")`. The client validates each SHA
is hex-only before sending; non-hex input is rejected without a network call.
The warm-cache path in `src/git/notes_api.rs::warm_cache_for_remote` chunks
SHAs into batches of **100** per request.

- Success response (200):

```ts
{
  notes: { [commit_sha: string]: string }
}
```

A 404 response is treated as success with an empty `notes` map ("no notes
found for any of the requested SHAs"). Any other non-200 status is surfaced
as an error and the warm-cache path treats it as a cache miss rather than a
hard failure.

## Metrics

### Upload Metrics

- Method: `POST`
- Path: `/worker/metrics/upload`
- Body:

```ts
{
  v: 1,
  events: Array<{
    t: number,
    e: number,
    v: { [position: string]: unknown },
    a: { [position: string]: unknown }
  }>
}
```

Event IDs:

- `1`: committed
- `2`: agent usage
- `3`: install hooks
- `4`: checkpoint
- `5`: session event

Common attribute positions:

| Position | Name |
| --- | --- |
| `0` | `git_ai_version` |
| `1` | `repo_url` |
| `2` | `author` |
| `3` | `commit_sha` |
| `4` | `base_commit_sha` |
| `5` | `branch` |
| `20` | `tool` |
| `21` | `model` |
| `22` | `prompt_id` (tombstoned; read only for legacy payloads) |
| `23` | `external_session_id` |
| `24` | `session_id` |
| `25` | `trace_id` |
| `26` | `parent_session_id` |
| `27` | `external_parent_session_id` |
| `30` | `custom_attributes` |

Committed event value positions:

| Position | Name |
| --- | --- |
| `0` | `human_additions` |
| `1` | `git_diff_deleted_lines` |
| `2` | `git_diff_added_lines` |
| `3` | `tool_model_pairs` |
| `4` | removed |
| `5` | `ai_additions` |
| `6` | `ai_accepted` |
| `7` | removed |
| `8` | removed |
| `9` | removed |
| `10` | `first_checkpoint_ts` |
| `11` | `commit_subject` |
| `12` | `commit_body` |
| `13` | `authorship_note` |
| `14` | `hunks` |

Install hooks event value positions:

| Position | Name |
| --- | --- |
| `0` | `tool_id` |
| `1` | `status` |
| `2` | `message` |

Checkpoint event value positions:

| Position | Name |
| --- | --- |
| `0` | `checkpoint_ts` |
| `1` | `kind` |
| `2` | `file_path` |
| `3` | `lines_added` |
| `4` | `lines_deleted` |
| `5` | `lines_added_sloc` |
| `6` | `lines_deleted_sloc` |
| `7` | `external_tool_use_id` |
| `8` | `edit_kind` |

Agent usage event values are currently an empty sparse map.

Session event value positions:

| Position | Name |
| --- | --- |
| `0` | `raw_json` |

| `2` | `external_parent_event_id` |

- Success response:

```ts
{
  errors: Array<{
    index: number,
    error: string
  }>
}
```

Partial errors are logged but not retried. 401 is treated as Unauthorized.

## Releases And Upgrade

### Fetch Releases

- Method: `GET`
- Path: `/worker/releases`
- Query string/body: none
- Success response:

```ts
{
  channels: {
    [channel: string]: {
      version: string,
      checksum: string
    }
  }
}
```

Known client-side channel names are:

- `latest`
- `next`
- `enterprise-latest`
- `enterprise-next`

### Download SHA256SUMS

- Method: `GET`
- Path: `/worker/releases/{channel}/download/SHA256SUMS`
- Path params:

```ts
{
  channel: string
}
```

- Success response: raw bytes.
- Expected body: UTF-8 text with lines in this format:

```text
<sha256 hash>  <filename>
```

The client verifies the raw bytes against the `checksum` from the releases
response before parsing the text.

### Download Install Script

- Method: `GET`
- Path: `/worker/releases/{channel}/download/{script_name}`
- Path params:

```ts
{
  channel: string,
  script_name: "install.ps1" | "install.sh"
}
```

`install.ps1` is used on Windows; `install.sh` is used elsewhere.

- Success response: raw bytes.
- Expected body: UTF-8 script text.

The client verifies the raw bytes against the matching filename entry from
`SHA256SUMS`.

## External HTTP Requests

These requests are also made by Rust code under `src/`, but they target external
services rather than the Git AI product API.

### GitLab Merge Requests

- Method: `GET`
- URL: `${CI_API_V4_URL}/projects/${CI_PROJECT_ID}/merge_requests`
- Query string:

```ts
{
  state: "merged",
  updated_after: string,
  order_by: "updated_at",
  sort: "desc",
  per_page: 100
}
```

`updated_after` is formatted as UTC `%Y-%m-%dT%H:%M:%SZ`.

- Headers:

```ts
{
  "PRIVATE-TOKEN": string
}
```

when `GITLAB_TOKEN` is set, otherwise:

```ts
{
  "JOB-TOKEN": string
}
```

when `CI_JOB_TOKEN` is set.

- Success response:

```ts
Array<{
  iid: number,
  title?: string,
  source_branch: string,
  target_branch: string,
  sha: string,
  merge_commit_sha?: string,
  squash_commit_sha?: string,
  squash?: boolean
}>
```

### JetBrains Marketplace Plugin Download

- Method: `GET`
- URL: `https://plugins.jetbrains.com/pluginManager`
- Query string:

```ts
{
  action: "download",
  id: string,
  build: `${product_code}-${build_number}`
}
```

- Success response: raw ZIP bytes.
- 404 is reported as plugin not found.

### PostHog Capture

- Method: `POST`
- URL: `${POSTHOG_HOST}/capture/`
- Default host: `https://us.i.posthog.com`
- Body:

```ts
{
  api_key: string,
  event: string,
  properties: {
    os: string,
    arch: string,
    version: string,
    message: string,
    level: string,
    [context_key: string]: unknown
  },
  distinct_id: string,
  timestamp: string
}
```

- Response: ignored.

### Sentry Store

- Method: `POST`
- URL derived from Sentry DSN:

```text
{scheme}://{host}/api/{project_id}/store/
```

- Headers:

```ts
{
  "X-Sentry-Auth": string,
  "Content-Type": "application/json"
}
```

- Body:

```ts
{
  message: string,
  level: string,
  timestamp: string,
  platform: "other",
  tags: { [key: string]: unknown },
  extra: { [key: string]: unknown },
  release: string
}
```

- Success response: any 2xx status.

## Source Map

- `src/auth/client.rs`: OAuth device code, token polling, refresh token, install nonce.
- `src/api/bundle.rs` and `src/api/types.rs`: bundle creation and payload types.
- `src/api/cas.rs` and `src/api/types.rs`: CAS upload/read endpoints and payload types.
- `src/api/metrics.rs`, `src/metrics/types.rs`, `src/metrics/events.rs`, and `src/metrics/attrs.rs`: metrics upload wire format.
- `src/commands/upgrade.rs`: releases and upgrade downloads.
- `src/ci/gitlab.rs`: GitLab merge request lookup.
- `src/mdm/jetbrains/download.rs`: JetBrains plugin ZIP download.
- `src/daemon/telemetry_worker.rs`: Sentry and PostHog dispatch, plus daemon-side metrics/CAS flushing.
